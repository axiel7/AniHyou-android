Current modules:

  :app                        → main Android application
  :baselineprofile            → baseline performance profile
  :core:base                  → constants (UNKNOWN_CHAR etc.)
  :core:common                → pure-JVM utilities (DateUtils, NumberUtils, StringUtils, ...)
  :core:network               → Apollo GraphQL + OkHttp, API modules
  :core:resources             → strings.xml and all XML resources (THE source of truth)
  :core:model                 → shared data models / GraphQL types

  :core:domain                → repositories, use-cases, Koin DI modules
  :feature:*                  → screen-level feature modules (activitydetails, calendar, ...)

## 4. BUILD RULES — NEVER BREAK THESE

### 4.1 String resources
- ALL R.string.* keys MUST exist in:
    core/resources/src/main/res/values/strings.xml
- The CI runs `bash scripts/check_string_refs.sh` BEFORE Gradle.
  Adding a Kotlin reference without adding the string = immediate CI fail.
- Script scans: app/ core/ feature/ wearos/

### 4.2 Module boundaries
- :wearos has its own wearCompileSdk / wearMinSdk (defined in root build.gradle.kts).
  Do NOT add :wearos as a dependency to any :app or :feature module.
- :core:base has no Android dependencies — keep it pure JVM.
- :core:common has no Android dependencies — keep it pure JVM.
- :core:network owns ALL network clients (Apollo, OkHttp, Retrofit/Ktor if added).
  Feature modules must NOT create their own HTTP clients.

### 4.3 Dependency injection
- DI framework: Koin (koin-android, koin-compose).
- All singletons go in:
    core/domain/src/main/java/.../core/domain/RepositoryModule.kt
    or a new *Module.kt in :core:domain — registered in App.kt startKoin{}.
- ViewModels: registered in app/.../ViewModelModule.kt.
- Never use Hilt or Dagger — wrong framework for this project.

### 4.4 Build flavors
The app has a "foss" product flavor. CI always builds assembleFossDebug.
If you add a new BuildConfig field, add it inside the foss flavor block too.

### 4.5 Kotlin / AGP versions
Defined in gradle/libs.versions.toml — do not hardcode version strings anywhere else.
Use alias(libs.plugins.*) in build.gradle.kts plugin blocks.
Use libs.* for dependency declarations.

### 4.6 WearOS
- :wearos is included in settings.gradle.kts — do NOT remove the include() line.
- Do NOT add or remove files from :wearos without verifying assembleFossDebug still passes
  (wearos is a separate app module; phone build does NOT compile wearos sources).

## 5. CI WORKFLOW
File: .github/workflows/build-pr.yml
Trigger: push to master, workflow_dispatch
Steps (in order):
  1. checkout
  2. setup JDK 17 (adopt distribution)
  3. bash scripts/check_string_refs.sh   ← fails fast on broken string refs
  4. ./gradlew assembleFossDebug         ← main build gate
  5. upload APK artifact

A build is only GREEN when BOTH steps 3 and 4 exit 0.

## 6. TESTING STRATEGY
Test files live at:
  app/src/test/java/com/axiel7/anihyou/          ← JVM unit tests (no emulator needed)
  app/src/androidTest/java/com/axiel7/anihyou/   ← instrumented tests (emulator required)

Key test files:
  BasicUnitTest.kt        → markdown formatting smoke test
  BuildIntegrityTest.kt   → guards StringUtils, NumberUtils, DateUtils at baseline commit
  DateTest.kt             → instrumented compose date rendering test

Rules for adding tests:
  - New feature added to master? → add a *Test.kt covering its pure-JVM logic.
  - Only add tests for NEW code introduced since the last GREEN build.
  - Never delete or weaken existing test assertions.
  - Tests for compose/UI go in androidTest; pure logic goes in test.
  - Run locally: ./gradlew test  (unit)  or  ./gradlew connectedDebugAndroidTest (instrumented)

## 8. COMMON FAILURE PATTERNS & FIXES

| Symptom                                      | Root cause                          | Fix |
|----------------------------------------------|-------------------------------------|-----|
| check_string_refs.sh: MISSING STRING         | R.string.X used in Kotlin, not in XML | Add to strings.xml |
| Unresolved reference: <ClassName>            | Wrong import or module not in deps   | Check module's build.gradle.kts dependencies |
| e: ... class 'X' is not abstract             | Missing override in sealed class impl | Implement all required members |
| Duplicate class found in modules             | Same class in two modules both on classpath | Remove duplicate, use single source |
| Cannot access 'X': it is internal in 'Y'    | Accessing internal across module boundary | Expose via public API or move to shared module |
| Merge conflict markers (<<<<<<< / =======)  | Un-resolved git merge                | Run git status, resolve, recommit |
| Base64 not found / wrong API level           | android.util.Base64 used in pure-JVM module | Use java.util.Base64 in :core:common/:core:base |
| Unresolved reference in *Test.kt (e.g. BasicUnitTest) | Import path stale after package refactor | Update import to current module path (e.g. core.ui.utils.* not plain utils.*) |
| Could not find androidx.wear:wear-phone-interactions in CI | :wearos dep unavailable; bare `./gradlew test` resolves ALL modules | Use `:app:testFossDebugUnitTest` scoped task instead |
| BuildIntegrityTest slugify tests fail (ComparisonFailure) | Test assertions assumed lowercase but StringUtils.slugify() preserves case | Match assertions to actual impl: "Re-Zero" not "re-zero"; use [a-zA-Z0-9\\-]+ regex |
| fatal: unable to access repo 403 on CI git push | GITHUB_TOKEN default is contents:read only | Add `permissions: contents: write` block to workflow YAML |
| Unresolved reference: icons / Icons in feature module | Missing compose material-icons dependencies | Add `libs.androidx.compose.material.icons.core` / `extended` to build.gradle.kts |
| Unresolved reference: datastore in feature module | Missing datastore preferences dependency | Add `libs.androidx.datastore.preferences` to build.gradle.kts |
| singleOf fails to resolve constructor ref | Class fails to compile or Koin generic resolution fails | Fix the class's compile errors or register using `single { ... }` block |
| toDoubleLocaleInvariant parses valid number FAILED | defaultNumberFormat.parse() throws ParseException on host machines with European locales | Implement `toDoubleLocaleInvariant` using Kotlin's locale-invariant `toDoubleOrNull()` |
| MainNavigation compile failures (removeLastOrNull) | Navigation3 lambda parameter is route object (not NavEntry) and backstack lacks removeLastOrNull | Access route properties directly and call topLevelBackStack.removeLast() |

## 10. WHAT TO DO WHEN YOU SEE A BUILD LOG
1. Check builds/logs.txt for the last GREEN entry.
2. Run diff between that commit and current HEAD.
3. Match the diff against Section 8 failure patterns.
4. Fix the root cause — do not comment out tests or downgrade checks.
5. After fix, append a new entry to builds/logs.txt.
